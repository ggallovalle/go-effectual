## Context

The `std/serde` module provides URL query string handling for both Go and Lua. Currently:

- `query.go` defines the `Query` struct with methods, some annotated (`//lua:nil-map`, `//lua:force-method`, `//lua:skip`)
- `query_bindings.go` is generated by lua-bindgen and contains instance method bindings
- `mod_query.go` is manually written and contains:
  - Module-level functions (`new`, `deserialize`, `serialize`) as Lua wrappers
  - Metamethod implementations (`__tostring`, `__pairs`)
  - Hardcoded annotation template
  - Full `ModQuery` struct implementing `LuaModDefinition`

This split requires maintaining duplicate knowledge - what `query.go` declares via annotations must match what `mod_query.go` manually implements.

## Goals / Non-Goals

**Goals:**
- Make `query.go` the single source of truth for all Query bindings
- Delete `mod_query.go` entirely
- lua-bindgen generates complete module bindings including registration
- Existing tests pass without modification
- `url.go` continues working (uses `NewQuery()` etc. directly)

**Non-Goals:**
- Modify `LuaMod[T]` interface semantics
- Change how other modules (slog, path, url) work
- Generate `QueryModuleApi` (not currently used by consumers)

## Decisions

### Decision: New annotation syntax

**Chosen approach:**
```go
//lua: class Query
//lua: module-fn new
//lua: metamethod __tostring
//lua: raw
```

**Alternatives considered:**
- `//lua-bindgen:class` - reuses existing annotation prefix
- `//lua:module-type Query` - more verbose

**Rationale:** Simple, readable, consistent with existing `//lua:` annotations.

### Decision: Module functions as package-level Go functions

**Chosen approach:**
```go
//lua: module-fn new
func NewQuery() *Query { ... }
```

**Alternatives considered:**
- Methods on a `QueryModule` struct (requires adding struct type, methods receive struct as receiver)

**Rationale:** Keeps existing `NewQuery()` for Go consumers (`url.go`). lua-bindgen wraps the package function for Lua.

### Decision: Verbatim metamethods with `//lua: raw`

**Chosen approach:**
```go
//lua: metamethod __pairs
//lua: raw
func QueryPairs(l *lua.State) int {
    // full implementation verbatim
}
```

**Alternatives considered:**
- DSL/builder pattern to compose metamethod behavior
- Separate annotation for behavior (e.g., `//lua: metamethod pairs,sort-first`)

**Rationale:** `__pairs` requires sort-then-iterate logic that cannot be auto-generated. Raw gives full control when needed.

### Decision: `LuaModOpen` split

**Chosen approach:**
```go
func LuaModOpen(l *lua.State, mod LuaModDefinition) {
    mod.OpenLib(l)
}

func LuaModOpenWithApi[T any](l *lua.State, mod LuaMod[T]) T {
    mod.OpenLib(l)
    return mod.Api(l)
}
```

**Alternatives considered:**
- Make `LuaModDefinition` include `Api()` method returning `any`
- Generic `LuaModOpen[T]` that works when `T` is interface{}

**Rationale:** Tests call `LuaModOpen` and discard the return. Making the simple case non-generic avoids type gymnastics.

## Risks / Trade-offs

- [Risk] Generated `Annotations()` must exactly match what tests expect → Mitigation: Generated template mirrors existing hardcoded template structure
- [Risk] `Pairs()` implementation is complex and must stay in sync → Mitigation: `//lua: raw` gives full control; documented pattern for similar cases
- [Risk] `url.go` depends on package-level functions → Mitigation: These remain unchanged; only binding layer moves to annotations

## Open Questions

- Should `//lua: metamethod` auto-register the metamethod in the metatable, or require explicit registration via generated `QueryLibrary()`?
- How do we generate the `__pairs` iterator function name to avoid collision with potential user methods named `Pairs`?
